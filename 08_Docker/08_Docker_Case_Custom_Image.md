# Тема 8: Практичний кейс — перехід від готового Docker-образу до власного

> **Матеріал для заняття.** Побудований на реальному кейсі з  
> `10_Implementation/openclaw/dev-prod-env-project`.

---

## 🎯 Навіщо цей кейс у Темі 8

У Темі 8 студентам важливо побачити не лише запуск готового контейнера з Docker Hub, а і момент, коли команді вже **недостатньо чужого образу** і потрібно створити **власний Docker image**.

Саме це і сталося в реальному кейсі `openclaw/dev-prod-env-project`.

---

## 1. Вихідна ситуація

На старті проєкт використовував готовий upstream-образ:

```yaml
services:
  openclaw-gateway:
    image: ghcr.io/openclaw/openclaw:latest
```

Тут важливо одразу пояснити, що таке `ghcr.io`.

`ghcr.io` — це **GitHub Container Registry**, тобто реєстр контейнерних образів від GitHub.  
Простими словами: це сховище, де лежать Docker-образи, звідки їх можна завантажити командою `docker pull` або використати в `docker-compose.yml`.

Розбір адреси:

```text
ghcr.io/openclaw/openclaw:latest
│       │        │        │
│       │        │        └── тег образу
│       │        └────────── назва образу
│       └─────────────────── власник / організація
└─────────────────────────── registry
```

Тобто:

- `ghcr.io` — де зберігається образ;
- `openclaw` — чий це простір у registry;
- `openclaw` — назва самого image;
- `latest` — тег, тобто мітка версії.

Це було зручно для швидкого запуску, але з часом з'явилась нова потреба:

- усередині контейнера потрібен `rsync`;
- потрібен `python3-pip`;
- потрібно встановити Python-залежності з `requirements.txt`;
- усе це мало відтворюватися автоматично і однаково в dev та prod.

---

## 2. У чому була проблема

Тимчасове рішення виглядало так: після старту контейнера Ansible виконував команди на кшталт:

```bash
docker compose exec openclaw apt-get install python3-pip
docker compose exec openclaw pip install -r requirements.txt
docker compose exec openclaw apt-get install rsync
```

Для заняття це дуже хороший приклад, бо він показує типовий анти-патерн.

### Чому це погано

- Ми змінюємо **живий контейнер**, а не описуємо його стан у Dockerfile.
- Після пересоздання контейнера ці зміни можуть зникнути.
- Такий підхід погано масштабується і погано перевіряється.
- Ansible починає "латати" контейнер після запуску замість того, щоб запускати вже готовий правильний образ.

Ключова думка для студентів:

> Якщо після `docker compose up` нам ще потрібно "доробляти" контейнер вручну або через `docker exec`, значить образ описаний неправильно.

---

## 3. Правильне рішення

Команда перейшла до підходу:

1. Беремо офіційний базовий образ.
2. Створюємо **власний Dockerfile** поверх нього.
3. Додаємо потрібні пакети та Python-залежності на етапі build.
4. Публікуємо власний образ у registry.
5. У `docker-compose.yml` використовуємо вже **свій image**, а не patching контейнера після старту.

Після цього Compose почав використовувати вже не upstream-образ, а кастомний:

```yaml
services:
  openclaw-gateway:
    image: ghcr.io/sobol-mo/openclaw-custom-image:latest
    pull_policy: always
```

---

## 4. Спрощена версія Dockerfile для показу на занятті

Нижче не "файл з продакшну один в один", а **навчальна адаптація** реального рішення з цього кейсу:

```dockerfile
FROM ghcr.io/openclaw/openclaw:latest

USER root

RUN apt-get update && apt-get install -y --no-install-recommends \
    rsync \
    python3-pip \
 && rm -rf /var/lib/apt/lists/*

WORKDIR /opt/openclaw-custom

COPY requirements.txt .

RUN pip3 install --no-cache-dir -r requirements.txt

USER node
```

### Що тут важливо проговорити

- `FROM` означає: ми не будуємо все з нуля, а розширюємо вже існуючий образ.
- `RUN` додає новий шар файлової системи.
- `COPY requirements.txt .` теж створює окремий шар.
- `RUN pip3 install ...` створює ще один шар.
- Фінальний контейнер запускається вже з усім готовим усередині.

## 5. ASCII-схема: як додаються шари образу

### Було: тільки upstream-образ

```text
┌─────────────────────────────────────────┐
│ Layer 1: ghcr.io/openclaw/openclaw      │
│ Базовий готовий образ                   │
└─────────────────────────────────────────┘
```

### Стало: власний образ поверх базового

```text
┌─────────────────────────────────────────┐
│ Layer 4: pip install -r requirements.txt│
│ Python-залежності для нашого кейсу      │
├─────────────────────────────────────────┤
│ Layer 3: COPY requirements.txt          │
│ Файл із переліком залежностей           │
├─────────────────────────────────────────┤
│ Layer 2: apt-get install rsync pip      │
│ Додаємо системні пакети                 │
├─────────────────────────────────────────┤
│ Layer 1: ghcr.io/openclaw/openclaw      │
│ Базовий upstream-образ                  │
└─────────────────────────────────────────┘
```

### Логіка для пояснення

```text
Базовий образ
   +
системні пакети
   +
наш requirements.txt
   +
встановлені Python-залежності
   =
наш власний готовий образ
```

## 6. Що саме змінюється в процесі роботи

### Варіант "неправильно"

```text
docker compose up
        │
        ▼
контейнер стартує
        │
        ▼
ansible/docker exec щось доставляє всередину
        │
        ▼
контейнер "живе", але його стан вже не дорівнює опису в Git
```

### Варіант "правильно"

```text
Dockerfile
   │
   ▼
docker build
   │
   ▼
готовий custom image
   │
   ▼
docker compose up
   │
   ▼
контейнер стартує вже в правильному стані
```

## 7. Як це пояснити студентам просто

Зручна формула для заняття:

- **готовий образ** підходить, коли нам вистачає стандартної комплектації;
- **власний образ** потрібен, коли проєкт має свої залежності;
- **Dockerfile** є "рецептом";
- **image** є "зібраним результатом";
- **container** є запущеним екземпляром цього результату.

Можна дати коротку аналогію:

```text
Dockerfile = рецепт
Image      = готова страва в коробці
Container  = страва, яку вже відкрили і подали
```

## 11. Місток до наступної теми: де тут з'являється CI/CD

Після того як студенти зрозуміли, **навіщо** потрібен власний image, природно виникає наступне питання:

> Хто і де буде його збирати щоразу, коли змінився Dockerfile або базовий образ?

Саме тут Docker переходить у тему **CI/CD**.

### Коротка відповідь

У реальному проєкті збірка нового image не повинна бути ручною дією на ноутбуці розробника. Її бере на себе **automation pipeline**, наприклад GitHub Actions.

Логіка виглядає так:

```text
Змінився Dockerfile або базовий image
            │
            ▼
GitHub Actions workflow запускається
            │
            ▼
docker build нового custom image
            │
            ▼
docker push у registry
            │
            ▼
docker-compose.yml / deploy бере вже новий image
```

---

## 12. Як це виглядає в цьому кейсі

У `dev-prod-env-project` зафіксована така ідея автоматизації:

- є окремий custom image на базі `ghcr.io/openclaw/openclaw:latest`;
- для нього передбачений GitHub Actions workflow `build.yml`;
- workflow виконує роль **watchdog**;
- він відстежує, чи не змінився upstream-базовий image;
- якщо зміна є, pipeline має ініціювати перебудову custom image;
- новий образ публікується в registry;
- далі середовище може підтягнути оновлений образ через `pull_policy: always`.

Для пояснення студентам це можна спростити до однієї фрази:

> Dockerfile описує, яким має бути образ, а CI/CD автоматично стежить, коли цей образ треба перебудувати і опублікувати знову.

---

## 13. Що саме перевіряє automation

У цьому кейсі важливий не лише rebuild від зміни власного коду, а і rebuild від зміни **базового образу**.

Ідея така:

```text
Наш custom image = наші шари + upstream base image
```

Отже, якщо upstream-образ оновився:

- у ньому можуть бути security fixes;
- у ньому можуть бути нові версії пакетів;
- у ньому можуть бути зміни, які нам треба перевірити;
- наш custom image теж потенційно застарів.

Тому pipeline перевіряє не "версію" в людському сенсі, а технічний ідентифікатор образу:

```text
Docker-Content-Digest
```

Це зручно проговорити на занятті так:

- тег `latest` зручний для читання;
- але для автоматизації важливіше не ім'я тегу, а **конкретний digest**;
- digest показує, що в registry реально лежить вже інший вміст образу.

## 14. Поточна задача проєкту: перевірка базового image і нотифікація розробника

Одна з актуальних задач цього проєкту якраз і полягає в такому ланцюжку:

1. За розкладом запускається GitHub Actions workflow.
2. Він отримує `Docker-Content-Digest` для upstream-образу.
3. Порівнює його з останнім збереженим значенням у файлі `openclaw-image-version`.
4. Якщо digest змінився:
5. Надсилає повідомлення розробнику в Telegram DevOps-канал.
6. Фіксує новий digest у репозиторії.
7. Це стає сигналом, що custom image треба перевірити і перебудувати.

ASCII-схема для пояснення:

```text
GitHub Actions (cron, кожні 6 годин)
                │
                ▼
   перевірка digest базового image
                │
       ┌────────┴────────┐
       │                 │
       ▼                 ▼
 digest той самий     digest змінився
       │                 │
       ▼                 ▼
   нічого не робимо   Telegram notification
                         │
                         ▼
              commit нового openclaw-image-version
                         │
                         ▼
               rebuild / deploy нового custom image
```

## 15. Як це пов'язати з темою CI/CD на занятті

Зручний перехід між темами можна сформулювати так:

> У Темі 8 ми вчимося створювати власний Docker image.  
> У Темі 9 ми додаємо automation, щоб цей image збирався, перевірявся і публікувався без ручної рутини.

Що варто сказати студентам:

- Docker без CI/CD часто залишається напівручним процесом;
- CI/CD робить збірку образів повторюваною;
- pipeline може реагувати не лише на `git push`, а і на зміну upstream image;
- нотифікація розробника є частиною DevOps-процесу, бо система має не лише виконувати дію, а і повідомляти про зміни.

Коротка формула:

```text
Dockerfile = що збирати
GitHub Actions = коли і як збирати
Registry = де зберігати
Notification = як дати знати команді, що щось змінилося
```
